# 知识库工程整体说明

## 1. 项目定位

本工程已经形成一套面向专业资料整理、知识组织与证据检索的结构化知识系统。它不是简单的文档存储系统，也不是只做概念管理的本体库，而是把原始资料、本体框架、证据挂接、索引体系和检索能力连成了一条完整链路，使系统能够围绕“知识节点”和“证据片段”提供可追溯的检索结果。

从当前能力看，这套系统主要解决三件事情：

1. 将原始资料整理为可计算、可检索、可扩展的结构化知识片段。
2. 将领域本体接入系统，形成稳定的知识框架、术语体系和关系网络。
3. 面向自然语言问题，返回围绕知识节点组织的证据包，而不是只返回若干文本命中结果。

因此，这个系统承担的是“知识底座”的职责：它负责把资料整理好、把知识框架建起来、把证据找出来并组织好；至于上层业务如何解释、裁剪或使用这些结果，不属于当前系统的职责范围。

---

## 2. 系统解决的核心问题

### 2.1 原始资料可阅读，但不具备稳定检索能力

业务资料往往来源复杂，格式不统一，既可能是纯文本，也可能是 Markdown、HTML 或其他导出文本。虽然这些资料可以人工阅读，但对系统而言，缺少稳定的结构信息，例如：

- 哪些内容是标题
- 哪些内容是正文
- 哪些内容是步骤项
- 哪些内容属于同一小节
- 哪些内容之间存在前后顺序或补充说明关系

如果这些信息缺失，系统即使做了检索，也很容易退化为“命中若干文字片段”，很难形成可用的知识结果。

### 2.2 仅靠文本匹配，无法稳定支撑专业问答

传统检索更擅长找“字面相似的段落”，但很难稳定回答以下问题：

- 命中的段落究竟在说明哪个专业概念
- 这段内容是在讲定义、步骤还是条件
- 哪几段是当前问题真正需要的证据

为了解决这个问题，系统引入本体（Ontology，领域概念框架）作为知识框架，把文档片段与领域概念连接起来，使检索从“找文字”提升为“围绕知识节点找证据”。

### 2.3 检索结果缺乏组织，阅读和使用成本高

如果系统只返回若干段文字，使用者还需要自行判断：

- 哪段是核心答案
- 哪段是补充说明
- 哪段是前提条件
- 哪段只是相邻背景

当前系统已经把这部分工作前置，检索结果按“命中的知识节点、核心证据、扩展证据、上下文”组织为证据包，从而降低理解成本和后续处理成本。

### 2.4 检索不仅要能运行，还要具备持续增强空间

本工程的目标不是做一个只能演示的样例，而是在保证结构清晰的前提下，形成一个可以持续增强的知识检索底座。当前系统已经将以下基础链路打通：

- 文档整理
- 本体接入
- 证据挂接
- 多路检索
- 索引支撑

在这一基础上，后续无论是继续增强术语归一、增加词表能力，还是提升向量能力和检索策略，都具备清晰的演进路径。

## 3. 系统整体结构

从系统方案角度看，当前工程可以分为五个部分：

### 3.1 文档整理层
负责把原始资料解析、切片并关系化，形成系统可理解的结构化知识片段。

### 3.2 本体管理层
负责把领域概念、类型、别名和概念之间的关系接入系统，形成知识框架。

### 3.3 证据挂接层
负责把文档片段与本体节点连接起来，明确“哪个片段在说明哪个知识节点”，以及“这个片段对该节点扮演什么证据角色”。

### 3.4 索引与检索层
负责把文本、本体、词项、映射关系和向量组织成可检索的索引体系，并完成问题理解、多路召回和排序。

### 3.5 结果组织层
负责把检索结果组织为证据包，而不是简单返回若干文本片段。

这五部分前后衔接，构成了一条完整的知识处理链路。

## 4. 文档整理层：原始资料如何转化为结构化知识

文档整理层是整个系统的起点，也是后续本体挂接和检索组织能否成立的基础。

### 4.1 设计目标

这一层的目标不是“按长度切块”，而是尽量保留原始资料的结构语义。系统关心的不只是文本内容本身，还包括文本在原文中的角色和位置，例如：

- 这是标题还是正文
- 这是步骤项还是说明项
- 它属于哪个章节
- 它和前后内容是什么关系

只有这些信息被保留下来，后续才能做稳定的证据挂接和证据扩展。

### 4.2 解析策略：先统一为结构树，再下沉为片段

系统对不同格式的文档并不直接输出平铺片段，而是先统一生成一棵 **DocNode 结构树**，再由统一的构建逻辑下沉为片段和关系。

当前 DocNode 覆盖的节点类型包括：

- `DOCUMENT`（整篇文档）
- `SECTION`（章节或标题节点）
- `PARAGRAPH`（正文段落）
- `LIST`（列表容器）
- `LIST_ITEM`（列表项）
- `TABLE`（表格容器）
- `TABLE_ROW`（表格行）
- `NOTE`（说明、提示、注意事项）
- `CODE_BLOCK`（代码块或配置块）
- `SENTENCE`（句子级辅助单元）

这种设计的作用在于：

1. 不同来源格式先被归一到同一结构模型。
2. 后续切片、关系构建、检索扩展不再依赖具体文档格式。
3. 系统处理对象从“原始文本”转变为“结构化节点”。

### 4.3 切片策略：主切片、结构锚点、辅助粒度三层并存

当前系统采用三层切片策略，以兼顾检索粒度、结构保留和局部定位。

#### 主切片（主要证据单元）
主切片是检索与证据挂接的主要对象，包括：

- `PARAGRAPH`（正文段落）
- `LIST_ITEM`（列表项）
- `NOTE`（说明/注意项）
- `TABLE_ROW`（表格行）
- `CODE_BLOCK`（代码块/配置块）

这些片段通常具备相对独立的语义表达，适合作为知识证据的基本单元。

#### 结构锚点（结构定位单元）
结构锚点用于保留文档结构，但不直接作为主要证据对象，包括：

- `SECTION`（章节标题）
- `LIST`（列表容器）
- `TABLE`（表格容器）

它们的价值在于帮助系统理解片段所在的章节和组织位置。

#### 辅助粒度（细粒度定位单元）
系统当前还保留：

- `SENTENCE`（句子）

句子级单元主要用于辅助关系建立和局部定位，不替代主切片。

### 4.4 关系建模：不仅保存片段，还保存片段之间的关系

如果只有切片，没有关系，检索结果仍然会退化为孤立段落。因此，系统在文档整理阶段同步建立两类关系。

#### 结构关系（Structural Relations）
用于保留文档原本的组织结构，当前固定包括：

- `PARENT`（父子关系）：表示结构上的包含关系，例如章节到段落。
- `NEXT`（前后顺序关系）：表示阅读或执行顺序上的前后连接。
- `SIBLING`（同级并列关系）：表示同一层级下的并列片段。
- `BELONGS_TO_SECTION`（归属章节关系）：表示某个片段属于哪个章节。
- `HAS_SENTENCE`（句子附属关系）：表示段落与句子的从属关系。

#### 轻量语义关系（Light Semantic Relations）
用于表达片段之间较弱但有价值的语义联系，当前固定包括：

- `ELABORATION`（补充说明关系）：表示后一段对前一段做补充解释。
- `CONDITION`（条件关系）：表示某段内容构成另一段内容的前提或条件。
- `SEQUENCE`（步骤顺序关系）：表示操作流程中的先后步骤。

这里的“轻量语义关系”并不意味着复杂推理，而是在结构基础上做有限增强，使系统在后续检索时能够扩展相关证据，而不是只返回单条命中。

### 4.5 这一层的产出

文档整理层最终产出两类结果：

1. 结构化知识片段。
2. 片段之间的结构关系与轻量语义关系。

这两类结果共同构成后续本体挂接和检索扩展的基础。

## 5. 本体管理层：系统如何组织领域知识

### 5.1 本体的作用

本体（Ontology）在系统中承担的是“知识框架”角色。文档中的表述可以多样，但系统若要稳定理解和组织知识，就必须有一套相对固定的概念体系。

### 5.2 当前本体管理的内容

系统当前管理的本体信息至少包括：

- 本体图本身
- 本体类型（Type）
- 本体节点（Node）
- 节点别名（Alias）
- 节点描述（Description）
- 节点之间的关系（Edge）

因此，本体并不是简单的名词表，而是一套“概念 + 类型 + 别名 + 关系”的知识框架。

### 5.3 本体接入方式

系统当前采用结构化 JSON 的接入方式，而不是依赖某一种说明文档格式。这样做的意义在于：

- 导入协议明确
- 结构稳定
- 不依赖人工文档排版
- 可以由脚本、表格转换程序或其他系统直接生成

### 5.4 本体关系的管理方式

系统当前会把节点之间的关系作为显式数据保存，而不是只保留为说明文字。也就是说，系统不仅知道“有哪些节点”，还知道“节点之间怎么连”。这一步的作用主要有三点：

1. 为 query 侧节点消歧提供图约束。
2. 为后续证据扩展提供相邻节点线索。
3. 为检索结果组织提供邻近知识单元。

### 5.5 本体在检索中的作用

本体接入以后，系统可以稳定处理三类事情：

1. **术语归一**：不同说法归并到同一个节点。
2. **关系理解**：节点之间的关系不再依赖临时文本判断。
3. **节点驱动检索**：问题先映射到知识节点，再围绕节点反查证据。

## 6. 证据挂接层：系统如何把文本证据连接到本体节点

### 6.1 这一层解决的核心问题

系统不仅要知道“文档里有一段话”，还要知道“这段话在说明哪个本体节点”。这就是证据挂接层的核心职责。

### 6.2 当前挂接思路

当前系统采用“规则优先、模型辅助”的策略。

第一步，先根据下列信号召回候选节点：

- 标准名命中
- 别名命中
- 描述关键词命中
- 类型名称补充分
- 图邻居辅助加分

第二步，对候选节点进行打分和排序。

第三步，当候选之间差异不明显时，再调用大模型做消歧。

这意味着，大模型并不是全量参与，而是只处理规则难以稳定判定的难例。

### 6.3 挂接关系的保存方式

当前系统将片段与本体节点之间的关系保存为显式数据，而不是临时计算后丢弃。保存时使用的是固定关系：

- `EVIDENCE_FOR`（证据指向关系）：表示该片段可作为某个本体节点的证据。

也就是说，系统当前不是保存一个泛化的“提到（mentions）”关系，而是保存“某个片段是某个节点的证据”。

### 6.4 匹配方式的保存方式

系统会把当前片段是通过什么方式挂到节点上的一起保存。当前固定的匹配方式包括：

- `STANDARD_NAME`（标准名命中）：直接命中节点标准名。
- `ALIAS`（别名命中）：通过节点别名命中。
- `DESCRIPTION`（描述命中）：通过描述关键词或说明文字命中。
- `LLM`（模型消歧）：通过大模型在候选之间做最终消歧。

在 query 理解阶段，还额外存在一些检索内部使用的匹配方式，例如 `NODE_FTS`、`ALIAS_FTS`、`VECTOR`、`LLM_QUERY`。这些方式主要用于 query 到 node 的候选召回和消歧，不直接对应最终的 segment-node 挂接结果。

### 6.5 证据角色标注

片段和节点发生关联，并不意味着它承担的作用相同。当前系统会为证据片段标注角色，并把结果保存到 `kb_segment_ontology_link.segment_role`。也就是说，角色不是临时算完就丢，而是作为挂接结果的一部分持久化保存。

当前固定的证据角色包括：

- `DEFINITION`（定义说明）：用于解释概念、术语或机制含义。
- `CONFIG_STEP`（操作步骤）：用于表达操作过程、执行动作、实施步骤。这个名称带有当前领域特征，底层含义更接近“动作步骤”。
- `PRECONDITION`（前提条件）：用于表达适用条件、触发条件、进入条件。
- `IMPACT`（影响说明）：用于表达影响、后果、风险、收益等说明。
- `CAUTION`（注意事项）：用于表达注意、禁止、必须避免等内容。
- `EXAMPLE`（示例说明）：用于表达举例、示范、样例。

### 6.6 角色判定的当前实现方式

当前角色不是通过复杂模型统一抽出来的，而是通过一组固定规则判定后落库。判定规则主要来自文本触发词，当前固定的触发方式包括：

- 出现“注意、禁止、避免、不要、必须”时，优先判为 `CAUTION`（注意事项）。
- 出现“如果、条件、前提、满足、才可、才可以”时，优先判为 `PRECONDITION`（前提条件）。
- 出现“步骤、首先、然后、最后、配置、割接、迁移、验证”时，优先判为 `CONFIG_STEP`（操作步骤）。
- 出现“影响、导致、风险、中断、收益、后果”时，优先判为 `IMPACT`（影响说明）。
- 出现“例如、比如、如”时，优先判为 `EXAMPLE`（示例说明）。
- 出现“是、指、用于、表示、即”时，优先判为 `DEFINITION`（定义说明）。
- 如果没有命中上述规则，当前默认回落到 `DEFINITION`（定义说明）。

这意味着，当前的角色体系和角色判定规则都是固定的，不是配置化的，也不是可学习的。

## 7. 索引与检索层：系统如何把问题转换为证据结果

这一层是系统的核心运行层。它负责把“用户提了一个问题”转换成“围绕某个知识节点组织出来的一组证据”。

### 7.1 Query 理解：先判断问的是什么，再决定怎么找

当前系统不会直接把 query 丢给全文检索，而是先做一层 Query 理解。这个过程至少包括三件事情。

#### 7.1.1 Query 归一化

系统会先对问题做基础清洗，例如：

- 去掉多余空白
- 做紧凑化归一
- 利用本体词项索引扩展相关术语

这里用到的是 **本体词项索引（Ontology Term Index）**。当前系统固定保存的词项类型包括：

- `STANDARD_NAME`（标准名词项）
- `ALIAS`（别名词项）
- `TYPE_NAME`（类型名称词项）
- `TYPE_KEY`（类型编码词项）
- `DESCRIPTION_KEYWORD`（描述关键词词项）
- `RELATION_TYPE`（关系类型词项）

这些词项共同构成 query 标准化和 query-node 召回的基础。

#### 7.1.2 问题类型识别

系统当前至少区分五类问题，这一组类别是固定的：

- `DEFINITION`（定义类问题）：例如“什么是……”。
- `OPERATION`（操作类问题）：例如“怎么做……”。
- `CONDITION`（条件类问题）：例如“什么条件下可以……”。
- `IMPACT`（影响类问题）：例如“为什么不能……”“会有什么影响……”。
- `RELATION`（关系类问题）：例如“和什么有关”“包含什么”“属于什么”。

当前问题类型识别采用“规则优先、LLM 兜底”的方式：

- 当规则命中足够明确时，直接按规则分类。
- 当规则置信度不足时，调用大模型在这五类中做判断。

也就是说，当前 queryType 不是开放集合，而是固定五类。

#### 7.1.3 问题类型到目标证据角色的映射

系统不会在拿到 queryType 后直接全量搜片段，而是会先把问题类型映射成“优先找什么证据角色”。当前这组映射关系是固定写死的：

- `DEFINITION`（定义类问题）优先找：
  - `DEFINITION`（定义说明）
  - `EXAMPLE`（示例说明）

- `OPERATION`（操作类问题）优先找：
  - `CONFIG_STEP`（操作步骤）
  - `CAUTION`（注意事项）
  - `EXAMPLE`（示例说明）

- `CONDITION`（条件类问题）优先找：
  - `PRECONDITION`（前提条件）
  - `CAUTION`（注意事项）

- `IMPACT`（影响类问题）优先找：
  - `IMPACT`（影响说明）
  - `CAUTION`（注意事项）

- `RELATION`（关系类问题）当前优先找：
  - `DEFINITION`（定义说明）
  - `CONFIG_STEP`（操作步骤）
  - `PRECONDITION`（前提条件）

这一步的意义是：检索不再是“所有片段混合搜索”，而是“面向特定证据角色的定向检索”。

### 7.2 Query 到 Node：先理解在问哪个知识节点

在完成 query 标准化和 queryType 判定之后，系统会进一步判断：

- 用户在问哪个 node
- 可能问哪几个 node
- 主节点（primary node）是谁
- 辅节点（secondary nodes）是谁

当前 node 召回不是单一路径，而是多种信号叠加，主要包括：

1. 词项索引匹配（标准名、别名、描述关键词等）。
2. 本体 FTS 匹配。
3. 向量候选匹配。
4. 基于节点关系的图约束加分。
5. 必要时的 LLM 消歧。

也就是说，系统当前并不是“先搜文本再猜节点”，而是“先尽量理解节点，再围绕节点找证据”。

### 7.3 Candidate 召回：三条检索支路并行工作

在确定 query 的目标方向后，系统会并行走三条召回支路。

#### 7.3.1 Node 驱动召回

这是主路。流程是：

`query -> ontology node -> segment evidence`

也就是：先命中节点，再从 `kb_segment_ontology_link` 中反查该节点对应的证据片段。

#### 7.3.2 文本召回

这是兜底路。即使 query 没有很好地命中本体节点，也可以依靠文本索引直接找：

- segment 标题
- segment 内容
- section 标题

这一支路当前由 **segment FTS** 支撑。

#### 7.3.3 向量召回

这是增强路。它主要解决：

- 术语没有完全对上
- query 是隐式表达
- 片段与 query 语义接近但不是字面命中

当前向量索引的对象类型也是固定的，主要包括：

- `ONTOLOGY_NODE`（本体节点向量）
- `SEGMENT`（片段向量）

### 7.4 排序：不是只看文本像不像

系统最终排序的不是“某个片段和 query 有多像”，而是“query、node、segment 三者是否形成高质量匹配”。因此排序时综合考虑的主要因素包括：

- query 与 node 的匹配分
- node 与 segment 的挂接置信度
- segment 角色是否符合当前 queryType 的目标角色
- 文本召回分和向量召回分
- 图邻居一致性

这意味着，当前排序对象并不只是 segment，而是“围绕 node 组织的证据候选”。

---

## 8. 结果组织层：系统如何把证据整理成可用结果

### 8.1 结果不是若干片段，而是证据包

系统当前不会直接返回“命中的若干段文字”，而是会把结果组织为 **Evidence Pack（证据包）**。每个证据包围绕一个知识节点组织，主要包含：

- 命中的 node
- 核心证据（core segments）
- 扩展证据（expanded segments）
- 上下文（context segments）
- 相邻 node

### 8.2 核心证据、扩展证据、上下文的区别

#### Core Segments（核心证据）
指直接支撑当前节点、并且在排序上最靠前的关键证据段。

#### Expanded Segments（扩展证据）
指基于片段关系扩展出来的相关证据。当前扩展不是自由扩散，而是沿着固定关系类型有控制地扩展。

当前扩展时优先考虑的关系顺序固定为：

- `PARENT`（父关系）
- `CHILD`（子关系，系统内部由 `PARENT` 的反向视角推导得到）
- `BELONGS_TO_SECTION`（归属章节）
- `SECTION_MEMBER`（章节成员，系统内部扩展视角）
- `SIBLING`（同级并列）
- `NEXT`（相邻顺序）
- `ELABORATION`（补充说明）
- `CONDITION`（条件关系）
- `SEQUENCE`（步骤顺序）

同时，不同 queryType 会采用不同的扩展偏好：

- `OPERATION`（操作类问题）优先扩步骤关系和章节内相关步骤。
- `CONDITION`（条件类问题）优先扩条件关系和补充说明。
- `DEFINITION`（定义类问题）优先扩定义补充和示例说明。
- `IMPACT` / `RELATION`（影响类/关系类问题）当前采用更保守的一跳扩展。

#### Context Segments（上下文）
指核心证据前后紧邻的背景说明，用于帮助理解，不直接参与主证据判断。

### 8.3 相邻节点的作用

系统在返回结果时，还会带出与主节点直接相关的邻近节点。这些节点不是最终答案本身，但能帮助使用者理解当前问题所处的知识上下文，例如：

- 条件节点周边的机制节点
- 方法节点周边的场景节点
- 场景节点周边的方法节点

这样，结果就不再是“搜到一段话”，而是“围绕一个知识节点组织的一组证据”。

## 9. 索引体系：系统如何支撑高效检索

当前系统的索引体系主要包括四类。

### 9.1 Segment FTS（片段全文索引）

用于支持 segment 标题、内容和 section 标题的全文检索。当前通过 `kb_segment_fts` 实现，主要服务于文本召回支路。

### 9.2 Ontology FTS（本体全文索引）

用于支持本体节点名称、别名、描述和类型相关信息的全文检索。当前包括：

- `kb_ontology_node_fts`
- `kb_ontology_alias_fts`

### 9.3 Ontology Term Index（本体词项索引）

用于支撑 query 标准化、术语归一、别名归一和候选 node 召回。它不是全文索引，而是以“词项 + 类型 + 权重”的方式组织起来的归一词表。

### 9.4 Vector Index（向量索引）

用于支撑语义相近召回。当前系统将向量结果保存为显式索引数据，并区分：

- 本体节点向量
- 片段向量

当前实现阶段，向量索引更多承担“增强召回”的职责，而不是替代主路检索。

## 10. 数据流：系统是如何跑起来的

从数据流角度看，当前系统主要有两条主线。

### 10.1 建档主线

原始资料进入系统后，依次完成：

1. 文档解析与结构树构建。
2. 切片下沉，生成结构化片段。
3. 建立片段关系。
4. 导入本体图。
5. 将片段挂接到本体节点。
6. 构建文本、本体、词项和向量索引。

这一条主线完成后，系统就具备了检索所需的基础数据。

### 10.2 检索主线

检索请求进入系统后，依次完成：

1. Query 归一化。
2. 问题类型识别。
3. 目标证据角色映射。
4. Query 到本体节点的解析与消歧。
5. Node 驱动召回、文本召回、向量召回并行执行。
6. 候选结果统一排序。
7. 证据扩展与上下文补全。
8. 组织成证据包并返回。

因此，当前系统的检索并不是“搜到几个段落就结束”，而是经历了“理解问题—找到节点—找证据—组织结果”的完整过程。

## 11. 当前系统的边界

为了避免对系统职责产生误解，当前边界需要明确说明。

### 11.1 当前系统负责什么

当前系统负责：

- 文档整理
- 本体接入
- 证据挂接
- 问题理解
- 多路召回
- 结果排序
- 证据包组织

### 11.2 当前系统不负责什么

当前系统不负责：

- 业务层面的最终裁剪、取舍和展示策略
- 业务层面的决策判断
- 通用推理引擎式的复杂知识推演
- 面向所有领域的通用角色体系配置化

例如，系统当前会返回 topK 证据包，也会给出排序分和置信度，但“最终要展示多少条、是否进一步过滤、如何组合成最终业务答案”属于上层业务来决定，而不属于当前知识库层。

### 11.3 当前系统里哪些内容是固定写死的

从实现上看，当前系统已经把若干类别体系和映射关系固定了下来，主要包括：

- QueryType 固定为五类：`DEFINITION`、`OPERATION`、`CONDITION`、`IMPACT`、`RELATION`。
- SegmentRole 固定为六类：`DEFINITION`、`CONFIG_STEP`、`PRECONDITION`、`IMPACT`、`CAUTION`、`EXAMPLE`。
- QueryType 到 TargetRoles 的映射关系是固定写死的。
- 文档结构关系和轻量语义关系当前也是固定集合。
- 词项索引中的 term_type 也是固定集合。
- 向量索引中的 item_type 当前固定为本体节点和片段两类。

这些固定集合使系统当前具备稳定性，但也意味着它们还没有完全配置化。后续如果知识范围扩大或领域发生变化，这些固定集合是最需要继续抽象和配置化的部分。

---

## 12. 总结

当前这套工程已经不是简单的资料导入工具，而是一套围绕“文档—本体—证据—检索”构建起来的知识底座系统。

它的基本工作方式可以概括为：

1. 先把原始资料整理成结构化片段，并保存片段之间的结构与语义关系。
2. 再把领域本体接入系统，形成稳定的概念框架。
3. 然后把片段挂接到本体节点，保存证据角色和匹配方式。
4. 最后围绕 query 做问题理解、多路召回、统一排序和证据组织，返回围绕知识节点构成的证据包。

从系统成熟度看，它已经具备了“建档—挂接—索引—检索—组织”的完整主链。当前最重要的特点，不是单点能力有多复杂，而是这些能力已经被串成了一个可运行、可解释、可继续增强的整体。
